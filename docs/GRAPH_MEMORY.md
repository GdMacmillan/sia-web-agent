# Graph Memory

The agent's persistent, long-term memory system. Enables institutional
knowledge to accumulate across sessions — every learning, pattern,
decision, and idea the agent encounters can be stored, searched, and
linked into a growing knowledge graph.

Graph memory is core to the agent's purpose. Self-modification depends
on remembering what worked, what didn't, and why. Without persistent
memory the agent has to re-derive every conclusion on every run.

## What it provides

- **Entity storage** with rich metadata (context, tags, priority, status).
- **Semantic search** using word2vec embeddings for natural-language queries.
- **Topic extraction** via TF-IDF for automatic categorization.
- **Relationship linking** between related entities.
- **Lifecycle tracking** for status management.
- **Durable persistence** via the backend's storage engine.

## Architecture

The agent does not store memory itself. It speaks HTTP to a graph-memory
backend supplied by the host:

```
┌─────────────────────────────────────────────────────────────────┐
│                            Agent                                 │
│                                                                  │
│    src/tools/memory-tools.ts                                     │
│      └── HTTP/REST → graph-memory backend                        │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│              Graph-Memory backend (host-provided)                │
│                                                                  │
│  REST API · DSL parser · word2vec · topic extraction · storage  │
└─────────────────────────────────────────────────────────────────┘
```

The agent points at the backend via env (see [Configuration](#configuration)).
The reference backend implementation lives in the sia-web monorepo
(`packages/graph-memory`, a Go service on port 8080 backed by BadgerDB
LSM-tree storage with ZSTD compression and a write-ahead log). Any
service that honors the REST surface described in
[REST API reference](#rest-api-reference) is sufficient.

## Configuration

The agent reads these env vars at boot:

| Variable | Purpose | Default |
|---|---|---|
| `GRAPH_MEMORY_HOST` | Backend host. | `localhost` |
| `GRAPH_MEMORY_PORT` | Backend port. | `8080` |
| `GRAPH_MEMORY_API` | Full URL override (takes precedence over host+port). | unset |

If the backend is unreachable, memory tool calls fail at call time with
an HTTP error. The agent does not pre-validate connectivity at boot.

## Data model

### Entity structure

```typescript
interface Entity {
  // Core
  id: string;                     // generated by backend
  entity_type: string;            // learning, pattern, decision, idea, note, task
  title: string;                  // 3–8 words
  content: string;                // detailed description

  // Categorization
  context?: string;               // system area (e.g. "middleware", "tools")
  tags?: string[];                // searchable keywords
  priority: "low" | "medium" | "high";
  status: string;                 // active, completed, archived, blocked, in-progress

  // Relationships
  related_entity_ids?: string[];
  relationship_types?: string[];

  // Custom data
  metadata?: Record<string, any>;

  // Timestamps
  created_at: string;             // ISO-8601
  updated_at: string;             // ISO-8601
}
```

### Entity types

| Type | Purpose | Examples |
|---|---|---|
| `learning` | Knowledge gained from debugging or discovery. | "Word2vec falls back to topic search for unknown terms" |
| `pattern` | Recurring approaches or conventions. | "Middleware stack pattern for tool composition" |
| `decision` | Choices made with documented rationale. | "Chose metadata-only relationships over graph edges" |
| `idea` | Proposed improvements or future work. | "Add caching layer for semantic search" |
| `note` | General observations or context. | "Ripgrep path resolution fails under Next.js" |
| `task` | Work items requiring action. | "Update docs after API redesign" |

### Relationship types

| Relationship | Use case | Example |
|---|---|---|
| `RELATED_TO` | General connection (default). | Topically related entities. |
| `IMPLEMENTS` | Entity implements another. | Learning implements an idea. |
| `DEPENDS_ON` | Entity requires another. | Task depends on task. |
| `SIMILAR_TO` | Address similar problems. | Two patterns for the same issue. |
| `SUPERSEDES` | Replaces or obsoletes. | New decision replaces old. |
| `CAUSED_BY` | Resulted from another. | Fix caused by bug report. |
| `EXTENDS` | Builds upon another. | Pattern extends base pattern. |
| `ESTABLISHES` | Creates or defines. | Decision establishes pattern. |
| `VALIDATES` | Proves or confirms. | Test validates learning. |
| `CONTRADICTS` | Conflicts with another. | Finding contradicts assumption. |

## Tool reference

Memory tools are defined in `src/tools/memory-tools.ts` and re-exported
from `src/tools/index.ts`. The default tool set (`createStandardTools`
in `src/deep-agent-setup.ts`) wires **five** of them through to the
agent: `store_entity`, `retrieve_entity`, `search_entities`,
`list_entities`, `update_entity_status`. The remaining three
(`update_entity`, `promote_entities`, `traverse_graph`) are present in
the source — `update_entity` is exported but not in the standard tool
list; `promote_entities` and `traverse_graph` aren't exported from
`tools/index.ts` at all. To use them, wire them in by extending the
tool list passed to `createDeepAgent`.

### `store_entity`

Store any type of entity. Returns the generated ID plus relationship
suggestions based on semantic similarity to existing entities.

```ts
store_entity({
  entity_type: "learning",
  title: "Graceful fallback for unknown query terms",
  content: "When word2vec encounters unknown terms, fall back to topic-based matching using TF-IDF keywords.",
  context: "graph-memory",
  tags: ["semantic-search", "error-handling"],
  priority: "high",
  status: "completed",
  related_entity_ids: ["conv_abc123"],
  relationship_types: ["IMPLEMENTS"],
  metadata: { files_changed: ["dsl.go"] },
});
```

### `search_entities`

Natural-language search. Uses word2vec semantic similarity with TF-IDF
topic fallback.

```ts
search_entities({
  query: "authentication patterns",
  entity_type: "pattern",     // optional filter
  tags: ["security"],          // optional filter
  priority: "high",            // optional filter
  status: "active",            // optional filter
  limit: 10,
});
```

### `list_entities`

Browse with filtering. Returns entities grouped by type with counts.

```ts
list_entities({
  entity_type: "idea",
  context: "performance",
  status: "active",
  limit: 50,
});
```

### `retrieve_entity`

Get full details of a specific entity.

```ts
retrieve_entity({
  entity_id: "conv_1702305000000",
});
```

### `update_entity_status`

Track lifecycle changes. Use for marking work in-progress, completed,
blocked, or archived.

```ts
update_entity_status({
  entity_id: "conv_1702305000000",
  status: "completed",
  notes: "Implemented; reduced latency by 45%.",
});
```

### `update_entity` *(not in default tool set)*

Update any entity field. Heavier than `update_entity_status` — prefer
the status-only tool when status is all that's changing.

### `promote_entities` *(not in default tool set, not exported)*

Defined in `src/tools/memory-tools.ts`. Promotes 3+ raw entities to a
synthesized level, or 3+ synthesized entities to abstract, generating
synthesized content via LLM. Calls `POST /admin/promote` on the
backend. Not currently wired through `tools/index.ts`.

### `traverse_graph` *(not in default tool set, not exported)*

Defined in `src/tools/memory-tools.ts`. Walks the relationship graph
from a seed node along outgoing / incoming / both edges. Calls
`POST /graph/traverse`. Not currently wired through `tools/index.ts`.

## Semantic search

### How it works

1. **Query processing** — input is tokenized; stop words removed.
2. **Embedding generation** — word2vec computes vectors per term.
3. **Document matching** — cosine similarity compares query vector to
   stored entity vectors.
4. **Fallback chain** — if word2vec doesn't know enough query terms,
   the backend falls back to topic similarity.

```
User query
    │
    ▼
┌─────────────────────────────────┐
│ DSL: MATCH CONVERSATIONS         │
│      SEMANTIC "query"            │
│      THRESHOLD 0.5 LIMIT 10      │
└────────────┬────────────────────┘
             │
             ▼
┌─────────────────────────────────┐
│ Word2vec semantic search         │
│ • Tokenize query                 │
│ • Generate embedding             │
│ • Compare to stored embeddings   │
│ • Rank by cosine similarity      │
└────────────┬────────────────────┘
             │
    ┌────────┴────────┐
    │ Unknown terms?  │
    └────────┬────────┘
             │ Yes
             ▼
┌─────────────────────────────────┐
│ Topic-based fallback             │
│ • Extract topics from query      │
│ • Find similar topics in graph   │
│ • Return connected entities      │
└────────────┬────────────────────┘
             ▼
        Ranked results
```

### Word2vec limitations

The reference backend uses pre-trained GoogleNews embeddings (~3M words,
300-dimensional). The vocabulary does not include technical identifiers
(camelCase names, acronyms, library-specific terms). Queries containing
only out-of-vocabulary terms fall through to the topic-based path —
which is usually fine, but can return surprising results for very
specific technical queries.

## Topic extraction

The reference backend extracts topics from every stored entity via
TF-IDF:

1. **Tokenization** — text split into words; stop words removed.
2. **N-gram generation** — unigrams + bigrams extracted.
3. **TF-IDF scoring** — term frequency weighted by inverse document
   frequency.
4. **Top keywords** — highest-scoring terms become the entity's topics.

Topics are then stored as nodes in the graph with `HAS_TOPIC` /
`SIMILAR_TO` / `CONTAINS` edges, enabling graph traversal to find
related entities via shared topics.

## DSL query language

The reference backend exposes a small DSL for graph queries. The agent
typically does not write DSL directly (the tools wrap it), but it's
documented here so the agent can reason about what the tools translate
to.

```sql
-- Semantic search
MATCH CONVERSATIONS SEMANTIC "query text" THRESHOLD 0.5 LIMIT 10

-- Topic-based search
MATCH CONVERSATIONS SIMILAR TO "authentication patterns"

-- Recent entities
MATCH CONVERSATIONS RECENT LIMIT 20

-- By agent
MATCH CONVERSATIONS BY manager

-- Property filters
MATCH CONVERSATIONS WHERE context="performance"

-- Create node
CREATE NODE doc1:Document {title:"My Doc", content:"..."}

-- Create edge
CREATE EDGE rel1:HAS_TOPIC FROM doc1 TO topic1 {weight:0.85}

-- Query by type
MATCH nodes:Conversation
MATCH edges:HAD_CONVERSATION
```

## REST API reference

The endpoints below are the ones the agent's client + memory tools
actually call (verified by grep in `src/clients/graph-memory-client.ts`
and `src/tools/memory-tools.ts`). A host implementing a compatible
backend needs to expose at least this surface.

| Method | Path | Used by |
|---|---|---|
| `POST` | `/conversations` | `store_entity` |
| `POST` | `/graph/edges` | `store_entity` (when adding relationships) |
| `POST` | `/graph/query` | `search_entities`, `list_entities`, internal DSL queries |
| `GET` | `/graph/nodes/:id` | `retrieve_entity` |
| `PATCH` | `/graph/nodes/:id` | `update_entity`, `update_entity_status` |
| `POST` | `/graph/traverse` | `traverse_graph` (if wired) |
| `POST` | `/admin/promote` | `promote_entities` (if wired) |
| `GET` | `/health` | `client.isAvailable()` helper |

The reference backend exposes additional endpoints (graph stats, manual
save, list-via-GET, etc.) — useful for operators but not on the agent's
call path. Don't rely on them being present in alternative
implementations.

### Store entity

```bash
curl -X POST http://${GRAPH_MEMORY_HOST}:${GRAPH_MEMORY_PORT}/conversations \
  -H "Content-Type: application/json" \
  -d '{
    "agent_id": "agent-id-here",
    "user_input": "[learning] My title",
    "agent_output": "Full description content",
    "context": "graph-memory",
    "metadata": {
      "entity_type": "learning",
      "title": "My title",
      "content": "Full description",
      "tags": ["tag1", "tag2"],
      "priority": "high",
      "status": "active",
      "related_entity_ids": ["conv_abc123"],
      "relationship_types": ["IMPLEMENTS"]
    }
  }'
```

### Semantic query

```bash
curl -X POST http://${GRAPH_MEMORY_HOST}:${GRAPH_MEMORY_PORT}/graph/query \
  -H "Content-Type: application/json" \
  -d '{"query": "MATCH CONVERSATIONS SEMANTIC \"authentication\" THRESHOLD 0.5 LIMIT 10"}'
```

### Update status

```bash
curl -X PATCH http://${GRAPH_MEMORY_HOST}:${GRAPH_MEMORY_PORT}/graph/nodes/conv_1234 \
  -H "Content-Type: application/json" \
  -d '{
    "properties": {
      "status": "completed",
      "status_notes": "Successfully implemented"
    }
  }'
```

## Relationship linking

### Automatic suggestions

When `store_entity` runs, the backend:

1. Searches for semantically similar entities.
2. Filters to the same entity type (learning → learning, idea → idea).
3. Returns the top 5 suggestions.
4. Suggests creating links to relevant entities.

The agent then decides whether to follow up with an `update_entity`
call to write the relationships back.

### Manual linking

Add relationships at store time when you already know what you're
linking:

```ts
store_entity({
  entity_type: "learning",
  title: "Cache implementation results",
  content: "Implemented the caching layer...",
  related_entity_ids: ["conv_idea_caching", "conv_decision_cache_strategy"],
  relationship_types: ["IMPLEMENTS", "VALIDATES"],
});
```

### Storage shape

Relationships are stored in entity metadata in the reference backend
(not as separate graph edges) for:

- **Simplicity** — no backend schema changes per relationship type.
- **Flexibility** — relationship semantics can evolve.
- **Portability** — entities are self-contained.

Tradeoff: no graph-traversal queries on relationships in the reference
backend (yet). `traverse_graph` operates at the entity level.

### Best practices

1. **Search before storing** to find linking opportunities.
2. **Use specific relationship types** over generic `RELATED_TO`.
3. **Link across entity types** — learning → idea, decision → pattern.
4. **Update superseded entities** to `archived` status.
5. **Chain relationships** to build coherent knowledge graphs:

   ```
   Idea: "Add caching"
       │
       └─IMPLEMENTS─▶ Learning: "Cache reduces latency 45%"
                          │
                          └─VALIDATES─▶ Decision: "Use LRU cache"
                                            │
                                            └─ESTABLISHES─▶ Pattern: "Cache-first lookup"
   ```

## Integration patterns

The agent uses memory throughout its lifecycle:

1. **Before planning** — search for related past attempts.
2. **During execution** — reference successful approaches.
3. **After completion** — store learnings and outcomes (handled
   automatically by `knowledge-formation.ts` middleware).
4. **During evaluation** — update idea status based on results.

## Troubleshooting

### "Failed to connect to graph-memory"

```bash
# Verify the backend is running
curl http://${GRAPH_MEMORY_HOST}:${GRAPH_MEMORY_PORT}/health

# Check the env you exported
echo "${GRAPH_MEMORY_HOST}:${GRAPH_MEMORY_PORT}"
```

### "No results from semantic search"

- Query terms may not be in the word2vec vocabulary; the backend will
  fall back to topic-based matching.
- Try broader query terms.
- Fall back to `list_entities` with filters.

### "Entities not persisting"

Confirm the backend's storage is writable. The reference backend
provides immediate durability via WAL; other backends may buffer and
require a manual flush — consult their docs.

## Performance notes

- **Search complexity** — semantic search is O(n) on entity count in
  the reference backend. Vector-index work is on the roadmap.
- **Word2vec load** — initial model load takes 2–3 seconds, then cached.
- **Storage** — the reference backend's LSM-tree can exceed RAM with
  disk spillover; compression keeps disk usage compact.

## Roadmap

- Graph edge-based relationships (true edges, not metadata).
- Relationship traversal queries.
- Vector index for faster semantic search.
- Backup and replication support.
